# 子代理工作流规范

## 1. 背景与问题

opencode 的子代理（subagent）每次被调用时都在**全新的子会话**中启动，**不携带之前的对话历史**。这意味着一个 agent 无法"记住"前一次调用时讨论的内容，对于持续数周的需求调研、设计、开发等工作流程构成障碍。

## 2. 解决方案：两层记忆

| 层级 | 手段 | 解决的问题 |
|------|------|-----------|
| 短期记忆 | `task_id` 会话恢复 | 同一次工作会话中多次调用的上下文连续 |
| 长期记忆 | `docs/` 文件持久化 | 跨天跨次、跨越项目阶段的成果传递 |

两者互补，缺一不可。

---

## 3. task_id 机制

### 3.1 什么是 task_id

每次调用 `task()` 工具时，opencode 会创建一个子代理会话，并返回一个会话 ID（即 `task_id`）。

```
task(subagent_type: "product-manager", prompt: "...")
  → 返回 task id="ses_14d01aa55ffeIvTbaq2YNRqYNH"
```

### 3.2 如何恢复会话

后续调用时传入同一 `task_id`：

```
task(
  subagent_type: "product-manager",
  task_id: "ses_14d01aa55ffeIvTbaq2YNRqYNH",
  prompt: "继续之前的工作，现在做用户画像"
)
```

子代理恢复完整的对话历史，就像断点续传。

### 3.3 task_id 的持久化

`task_id` 由调用方（主代理）管理。本团队的规范是将 `task_id` 写入文件：

```
.opencode/.session-<agent-name>
```

每次调用流程：

```
检查文件 → 存在则读取 task_id → 调用 task(task_id) → 提取返回的 task_id → 写入文件
```

### 3.4 task_id 的生命周期

- `task_id` 对应的子代理会话存储在 `~/.local/share/opencode/opencode.db`（SQLite）
- 只要主会话未删除，子会话也保留
- 主会话可以通过 `opencode -s <session-id>` 恢复（参见第 5 节）

## 4. 文件持久化

### 4.1 目录约定

```
产品资料/
  0_需求/
    产品定位一页纸.md          ← product-manager 产出
    利益相关方登记册.md          ← product-manager 产出
    竞品分析报告.md              ← product-manager 产出
    ConDili_软件需求规格说明书.md        ← product-manager 产出（核心交付物）
    调研/
      访谈提纲/                 ← product-manager 产出
      访谈记录/                 ← product-manager 产出
      待确认需求清单.md          ← product-manager 产出
  1_概要设计/
    ConDili_软件概要设计说明书.md        ← architect 产出
  2_详细设计/
    ConDili_软件详细设计说明书.md        ← developer 产出
  3_程序代码/                          ← developer 产出
  4_测试/
    ConDili_测试方案.md        ← tester 产出
    ConDili_测试案例.md        ← tester 产出
```

### 4.2 agent 工作模式

每个 agent 在 prompt 中均应包含三段指令：

```
1. 读取已有文件：启动时扫描 docs/ 目录，理解上下文
2. 执行本职工作：基于已有材料完成本次任务
3. 写入产出：将成果写入 docs/ 下对应的文件
```

### 4.3 模板文件

项目 `rules/` 目录下存放各阶段文档的模板：

| 文件 | 用途 |
|------|------|
| `rules/《软件需求规格书说明书》模板.md` | 产品经理使用 |
| `rules/《软件概要设计说明书》模板.md` | 架构师使用 |
| `rules/《软件详细设计说明书》模板.md` | 详细设计师使用 |

## 5. 主会话管理（面向用户）

### 5.1 建议用一个会话贯穿整个项目

```
# 第一次启动
opencode

# 后续恢复（推荐）
opencode -c                    # 恢复最近一次会话

# 或指定会话
opencode session list           # 列出所有会话
opencode -s "ses_xxx"          # 恢复指定会话
```

### 5.2 多次任务间的典型操作

```
你 → @product-manager  "开始需求调研，先做产品定位"
     → agent 产出 docs/产品定位一页纸.md

（几天后，opencode -c 恢复主会话）

你 → @product-manager  "继续调研，基于定位做用户画像"
     → 主代理自动读取 task_id，恢复同一会话
     → agent 有完整对话历史 + 已读 docs/ 文件，继续工作
     → 产出 docs/利益相关方登记册.md
```

## 6. agent 创建规范

### 6.1 agent 文件位置

```
.opencode/agents/<name>.md
```

### 6.2 agent 文件结构

```markdown
---
description: 一句话描述 agent 的职责和适用场景
mode: subagent
permission:    # 按需设置权限。
  read: allow
  write: allow
  edit: allow
  bash: deny
  glob: allow
  grep: allow
  webfetch: allow
  question: allow
  task: deny
  skill: deny
  todowrite: deny
model: <provider>/<model-id>
temperature: 0.1
---

# 角色定义
你是一位资深的<角色名称>，擅长……

# 核心职责
1. ……
2. ……
3. ……

# 工作流程
1. 启动时，先读取 docs/ 中的已有文件，理解上下文。
2. 基于已有材料执行本职工作。
3. 将成果写入 docs/ 对应的文件中。
```

### 6.3 权限说明

| 工具 | 产品经理 | 架构师 | 开发者 | 测试 |
|------|---------|-------|-------|------|
| read | √ | √ | √ | √ |
| write | √（写文档） | √（写文档） | √（写代码） | √（写测试） |
| edit | × | × | √ | × |
| bash | × | × | √（运行测试） | √（运行测试） |

## 7. 团队共享方案

整个工作流规范通过以下文件纳入 git 管理，团队共享：

| 文件 | 作用 |
|------|------|
| `.opencode/skills/team-workflow/SKILL.md` | 技能文件，教会主代理自动管理 task_id |
| `.opencode/agents/<name>.md` | 各子代理的定义和行为指令 |
| `docs/` | 项目文档和工作产出 |
| `rules/` | 各阶段文档模板 |
| `AGENTS.md` | 项目级通用指令 |

团队成员只需 `git pull`，opencode 自动发现所有技能和代理，无需个人配置。

## 8. 角色分工模型

推荐按"技术经理 + 多名工程师"的模式组织 agent：

```
你（人类）  ← 编排者，决定何时做什么
  │
  ├─ @product-manager     需求调研 → 软件需求规格说明书
  ├─ @architect            概要设计 → 软件概要设计说明书
  ├─ @developer      详细设计 → 软件详细设计说明书、程序代码
  └─ @tester               编写测试 → 测试方案、测试用例
```

每个 agent 专注自己的职责，通过git仓库中的文件传递信息。

对于小型项目，一个 developer agent 可以负责全部模块的编码。当项目规模增大时，可按模块拆分为 `@dev-auth`、`@dev-payment` 等，每个 agent 从共享的设计文档中读取接口规范后再编码。
